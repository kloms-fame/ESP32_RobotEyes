## 2. 不可变更的既定基线

以下内容已完成完整技术调研与论证，开发中必须严格执行，不得随意替换方案、推翻架构。

### 2.1 技术选型基线

| 领域 | 选定方案 | 约束说明 |
|------|----------|----------|
| 主控框架 | Arduino ESP32 框架 + 内置 FreeRTOS | 优先使用原生 FreeRTOS API |
| OLED 驱动 | U8g2 双实例 | 左眼 I2C SSD1306 / 右眼 SPI GME12864，独立渲染 |
| 舵机驱动 | ESP32Servo | 基于 LEDC 硬件 PWM，50Hz 标准信号 |
| 事件通信 | 基于 FreeRTOS Queue 的 EventBus | 模块间唯一合法通信方式 |
| Web 服务 | ESPAsyncWebServer | 异步非阻塞架构，支持 WebSocket |
| 眼动核心 | 参考 esp32-eyes 架构思想 | 双屏独立改造，保留平滑插值、微动作能力 |
| 表情资源 | 参考 Irisoled 位图资源 | 按标准表情结构体封装 |

### 2.2 软件架构基线

**严格遵循四层分层架构**：

│  │ WebConsole   │  │ Expression   │  │ BehaviorEngine       │  │
│  │ (ESPUI)      │  │ Manager      │  │ (情绪自动触发)        │  │
│  └──────┬───────┘  └──────┬───────┘  └─────────┬────────────┘  │
├─────────┼─────────────────┼─────────────────────┼───────────────┤
│                        核心服务层 (Core Service Layer)           │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────────────┐  │
│  │ 事件总线     │  │ 状态机       │  │ 配置管理             │  │
│  │ EventBus     │  │ StateMachine │  │ ConfigManager        │  │
│  │ (FreeRTOS    │  │ (表情切换/   │  │ (存储自定义参数)      │  │

│  └──────┬───────┘  └──────┬───────┘  └─────────┬────────────┘  │

| 违规行为 | 严重程度 |
|----------|----------|
| 跨层调用 | 严重违规 |
| 跨模块直连 | 严重违规 |
| 业务逻辑下沉到驱动层 | 严重违规 |

### 2.4 FreeRTOS 任务划分基线

| 任务名称 | 优先级 | 栈大小 | 职责 | 通信方式 |
|---------|--------|--------|------|---------|
| **InputTask** | 高(2) | 2048 | ADC键盘采样、摇杆采样、事件推送 | EventBus队列 |
| **RenderTask** | 中(1) | 4096 | 左眼渲染、右眼渲染、动画帧更新 | 共享数据+EventBus |
| **ServoTask** | 中(1) | 2048 | 舵机角度插值、眉毛微动作 | EventBus队列 |
| **WebTask** | 低(0) | 4096 | WiFi管理、HTTP服务、WebSocket | EventBus队列 |
| **BehaviorTask** | 中(1) | 2048 | Idle微动作、情绪状态机 | EventBus队列 |

**约束**：禁止随意新增任务或打乱职责边界。

## 3. 四大核心工作铁律

优先级高于一切开发需求，任何情况下必须遵守。

### 铁律 1：开源优先，绝对禁止重复造轮子

开发任何功能前，必须按以下优先级逐级决策，禁止直接进入自研：

| 优先级 | 复用方式 | 适用条件 |
|--------|----------|----------|
| 1 | 直接使用 | 已有成熟稳定开源库/模块，API完全满足需求 |
| 2 | Fork 修改 | 开源模块核心逻辑可用，仅需少量适配 |
| 3 | 借鉴设计 | 无直接可用代码，但有成熟架构/算法参考 |
| 4 | 自主实现 | 完全无对应开源方案 |

**规则**：但凡有经过社区验证的成熟实现，绝不允许从头编写同类功能。

### 铁律 2：架构红线不可突破

| 禁止行为 | 后果 |
|----------|------|
| 跨层调用（应用层直接操作硬件寄存器） | 严重违规 |
| 跨层调用（驱动层包含业务逻辑） | 严重违规 |
| 模块间直接函数调用 | 严重违规 |
| 修改已稳定模块的对外接口 | 严重违规 |

**规则**：
1. 所有跨模块交互只能通过 EventBus 事件队列
2. 扩展只能新增，不能破坏原有兼容性

### 铁律 3：面向结果交付，拒绝"代码写完即完成"

| 验收标准 | 要求 |
|----------|------|
| 功能交付 | 以可量化、可复现的实际硬件运行结果为标准 |
| 测试步骤 | 必须提供明确的测试步骤、预期现象、验收标准 |
| 独立验证 | 用户可按步骤独立验证功能正确性 |

**规则**：禁止以"代码逻辑正确"作为功能完成的判断标准。

### 铁律 4：渐进式开发，不超前、不越界

| 规则 | 要求 |
|------|------|
| 单次聚焦 | 单次对话仅聚焦当前指定的一个功能/模块 |
| 路线遵守 | 严格按照开发路线图顺序推进，不跳步 |
| 版本验收 | 每个稳定版本经用户验收通过后，本对话使命结束 |
| 新会话启动 | 下一功能在全新对话中启动 |

## 4. 编码前强制停止检查

在写下第一行代码之前，必须完整回答以下 10 个问题，全部通过后方可进入开发：

| 序号 | 问题 | 通过条件 |
|------|------|----------|
| 1 | 这个功能是否有成熟的开源实现？具体是哪个项目/哪个模块？ | 明确列出项目名称和模块路径 |
| 2 | 为什么不能直接复用？必须修改的点有哪些？修改工作量有多大？ | 明确修改原因和工作量评估 |
| 3 | 是否存在更简单、更稳定的替代方案？ | 评估后给出结论 |
| 4 | 实现方案是否违反项目分层架构？是否会造成模块耦合？ | 符合架构要求，无耦合 |
| 5 | 功能的输入、输出、接口是否明确？是否符合 EventBus 通信规范？ | 输入输出明确，通过 EventBus 通信 |
| 6 | 是否会引入新的魔法数字？是否所有可调参数都纳入 Config 管理？ | 无魔法数字，参数可配置 |
| 7 | 是否会使用阻塞式 delay()？是否都用任务延时/硬件定时器替代？ | 使用非阻塞方案 |
| 8 | 有没有对应的测试用例？验收标准是否可量化、可验证？ | 测试用例完整，标准可量化 |
| 9 | 是否会影响现有已稳定的功能？是否需要回归测试？ | 不影响或有回归测试计划 |
| 10 | 开源复用部分的协议是什么？是否符合项目开源合规要求？ | 协议明确，符合合规要求 |

| 7 | Unit Test | 单元测试报告 | 单元测试通过，覆盖边界条件 |

## 6. 代码质量强制规则

违反以下任何一条即为不合格交付。

### 6.1 绝对禁止项

| 序号 | 禁止内容 | 违规后果 |
|------|----------|----------|
| 1 | 单个 .cpp/.c 文件超过 500 行 | 必须拆分为独立子模块 |
| 2 | 在 loop()/主循环中编写业务逻辑 | 必须拆分到独立 Task |
| 3 | 使用阻塞式 delay() | 必须使用 vTaskDelay/硬件定时器/调度器 |
| 4 | 出现魔法数字 | 必须通过宏、枚举、配置常量定义 |
| 5 | switch-case 分支超过 30 行 | 必须使用状态模式实现 |
| 6 | 复制粘贴重复代码 | 必须抽取为公共函数/工具类 |
| 7 | 模块间直接调用对方的内部函数 | 必须通过 EventBus 事件交互 |
| 8 | 全局变量泛滥 | 必须通过互斥锁保护的结构体访问 |

### 6.2 必须遵守项

| 序号 | 要求内容 | 验收标准 |
|------|----------|----------|
| 1 | 所有可调参数统一纳入 Config 管理 | 支持运行时修改与持久化存储 |
| 2 | 所有输入统一封装为事件推入 EventBus | 按键、摇杆、Web 命令均通过事件传递 |
| 3 | 所有模块头文件只暴露对外接口 | 内部实现细节隐藏在 cpp 中 |
| 4 | 所有函数、关键变量必须有清晰的注释 | 命名见名知意，注释准确 |
| 5 | 所有动态内存分配必须考虑释放 | 无内存泄漏风险 |
| 6 | 所有硬件引脚、外设参数统一在配置文件中定义 | 不得散落在业务代码里 |

## 7. 功能规格与测试规范

### 7.1 Feature Spec 要求

每个功能必须在 `docs/features/` 下生成独立规格文档，包含且仅包含以下章节：

| 章节 | 内容要求 |
|------|----------|
| 功能目标与边界 | 明确功能做什么、不做什么 |
| 输入定义 | 所有输入信号、事件类型、参数范围 |
| 输出定义 | 所有输出信号、显示效果、舵机动作 |
| 对外接口 | 暴露的函数、事件、数据结构 |
| 测试用例 | 可执行的测试步骤 |
| 验收标准 | 可量化的验收条件 |
| 开源项目参考 | 参考的开源项目与模块 |
| 预估工作量 | 人天估算 |
| 完成定义 | 明确的完成标志 |

### 7.2 Test Spec 要求

所有测试必须面向结果、可观测、可复现：

**正确示例**：
- 摇杆推到最左 → LookTarget X 值为 -1.0 → 眼球移动到屏幕最左侧
- 触发眨眼指令 → 500ms 内完成闭眼再睁眼完整动作

**错误示例**：
- 代码逻辑正确（不可观测）
- 功能已实现（不可量化）

**规则**：禁止以"代码逻辑正确"作为测试通过标准。

## 8. 代码自审清单

每开发完一个功能，必须逐项回答以下问题，全部合格才可交付用户验收：

| 序号 | 自查问题 | 合格标准 |
|------|----------|----------|
| 1 | 是否存在重复代码？ | 无复制粘贴的重复逻辑 |
| 2 | 是否违反 SOLID 设计原则？ | 符合单一职责、开闭、依赖倒置等原则 |
| 3 | 是否存在重复造轮子的部分？ | 已充分检索开源方案 |
| 4 | 是否存在模块直接调用的情况？ | 所有跨模块交互通过 EventBus |
| 5 | 是否存在未定义的魔法数字？ | 所有常量已定义 |
| 6 | 是否存在内存泄漏风险？ | 动态分配均有释放机制 |
| 7 | 是否影响后续功能扩展？ | 接口设计支持扩展 |
| 8 | 是否遵守分层架构规范？ | 无跨层调用 |
| 9 | 是否所有事件都通过 EventBus 传递？ | 无直接函数调用 |
| 10 | 是否标注了所有开源复用的来源与协议？ | 来源明确，协议合规 |

## 9. 开源复用登记规则

每开发一个功能，必须在 `docs/OPEN_SOURCE.md` 中登记对应条目。

### 登记内容必须包含

| 字段 | 内容要求 |
|------|----------|
| 功能名称 | 清晰描述功能 |
| 参考项目/模块 | 明确项目名称和具体模块路径 |
| 复用方式 | 直接使用 / Fork 修改 / 借鉴设计 / 自主实现 |
| 修改内容 | 如果是修改复用，列出具体修改点 |
| 工作量 | 人天估算 |
| 开源协议 | 项目使用的开源协议 |
| 不自研的理由 | 说明选择开源方案的原因 |

### 登记流程

```
功能启动 → 开源方案评估 → 登记 OPEN_SOURCE.md → 编码开始
```

**规则**：未完成登记不得进入编码阶段。

## 10. 单次对话边界规则

| 规则 | 要求 |
|------|------|
| 单次聚焦 | 每次对话仅处理一个明确的功能点 |
| 范围控制 | 不主动扩展范围、不提前开发后续功能 |
| 迭代修复 | 功能未通过用户验收，本对话不结束，持续迭代 |
| 验收终止 | 功能验收通过、完成 Git 提交后，本对话终止 |
| 新会话启动 | 下一个功能必须开启全新对话，避免上下文污染与架构漂移 |
